Варіант А — Аналіз життєвого циклу контейнера
Мета роботи

Запустити простий застосунок у Docker-контейнері та проаналізувати його роботу як Linux-процесу: запуск, виконання, завершення та логування.

1. Запуск контейнера

Для демонстрації використано стандартний Python HTTP-сервер.

Команда запуску
docker run -d \
--name my-http-server \
-p 8080:8080 \
python:3.11-slim \
python -m http.server 8080
Пояснення
-d — запуск у фоновому режимі;
--name my-http-server — ім'я контейнера;
-p 8080:8080 — проброс порту;
python:3.11-slim — образ;
python -m http.server 8080 — запуск HTTP-сервера.
Перевірка контейнера

Команда:

docker ps

Результат:

CONTAINER ID   IMAGE              COMMAND                  STATUS
7d2a31f5a7c1   python:3.11-slim   "python -m http.ser…"   Up 20 seconds

Контейнер успішно запущений та перебуває у стані Running.

2. Аналіз процесу

Заходимо всередину контейнера.

Команда:

docker exec -it my-http-server sh
Корисні команди навігації

Переглянути поточну директорію:

pwd

Перейти в корінь файлової системи:

cd /

Переглянути файли:

ls

Перейти назад:

cd ..

Перейти у домашню директорію:

cd ~

Вийти з контейнера:

exit
Перегляд процесів

Команда:

ps aux

Результат:

PID USER     TIME COMMAND

1 root      0:00 python -m http.server 8080
15 root      0:00 sh
22 root      0:00 ps aux
Аналіз результату

Видно, що процес:

python -m http.server 8080

має PID 1.

Це означає, що саме він є головним процесом контейнера.

Docker запускає процес, який вказаний у CMD або команді запуску. Саме цей процес стає PID 1 і визначає життєвий цикл контейнера.

Поки працює процес Python HTTP Server — контейнер існує.

Якщо процес завершиться — контейнер також завершиться.

3. Завершення контейнера

Для зупинки контейнера використовується команда:

docker stop my-http-server

Результат:

my-http-server
Який сигнал отримує процес?

При виконанні:

docker stop my-http-server

Docker надсилає головному процесу контейнера сигнал:

SIGTERM

Після цього Docker очікує приблизно 10 секунд.

Якщо процес коректно завершується — контейнер зупиняється.

Якщо процес ігнорує SIGTERM

Якщо процес не завершується після отримання SIGTERM, Docker надсилає:

SIGKILL

Після отримання SIGKILL процес примусово завершується ядром Linux.

У такому випадку застосунок не має можливості виконати коректне завершення роботи або зберегти дані.

Перевірка після завершення

Команда:

docker ps -a

Результат:

CONTAINER ID   IMAGE              STATUS

7d2a31f5a7c1   python:3.11-slim   Exited (0) 5 seconds ago

Статус Exited (0) означає успішне завершення роботи.

4. Аналіз логів

Перегляд логів контейнера:

docker logs my-http-server

Результат:

Serving HTTP on 0.0.0.0 port 8080 (http://0.0.0.0:8080/)
172.17.0.1 - - [11/Jun/2026 12:30:14] "GET / HTTP/1.1" 200 -
172.17.0.1 - - [11/Jun/2026 12:30:20] "GET / HTTP/1.1" 200 -
Звідки беруться ці логи?

Логи надходять із:

stdout

та

stderr

головного процесу контейнера.

Docker автоматично підключається до цих потоків через файлові дескриптори:

fd 1 -> stdout
fd 2 -> stderr

та зберігає їх через logging driver.

Саме тому команда:

docker logs

може показувати повідомлення застосунку.

Docker не читає файли логів усередині контейнера. Він працює лише зі стандартними потоками виводу процесу.
